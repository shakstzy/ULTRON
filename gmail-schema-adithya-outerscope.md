## 5. Content-based heuristics

Each rule expressed as: "if <condition>, then <action>, because <reasoning>."

- **If sender display name has "Firstname Lastname" pattern AND domain is NOT a known noise platform AND subject starts with `Re:` or `Fwd:`, then mark as deserves-reply (UNREAD), because it's a real human reply chain.** (Used in `is_deserves_reply` heuristic during inbox classification.)

- **If `List-Unsubscribe` header is present, then the message is mass-mail, not a personal correspondence, regardless of sender display name.** Used as the primary noise discriminator. Senders WITH List-Unsubscribe go to Newsletters or trash. Without → Misc or substantive label.

- **If sender ends in `@gmail.com`, `@outlook.com`, `@hotmail.com`, `@yahoo.com`, `@icloud.com`, `@me.com`, `@proton.me`, then it's likely a real human (or Adithya's own account self-CC).** These are rescue candidates from any sweep.

- **If the message is in a thread with > 1 message AND at least one message lacks a calendar-invite subject pattern, then the thread is conversational — do NOT trash any thread member regardless of subject pattern.** Per round 14 fix; calendar trash filter now exempts `-subject:"Re:" -subject:"Fwd:" -subject:"FW:"`.

- **If a message has the same threadId as a labeled message, then apply the same label to maintain thread integrity.** Per round 13: every thread gets exactly one user label (highest priority among the union of labels across thread messages). Adithya's sent replies (which I labeled Personal) get re-labeled to match the thread's primary label (Hiring/BD/etc.).

- **If the message is from a `@docusign.net` sender AND subject contains "Verify a New Device" / "password" / "viewed", then it's account/security mail, NOT a contract.** Goes to Dev, not Contracts. Round 5 distinction.

- **If a Stripe receipt is from `notifications@mail.granola.ai` (Granola product marketing), then trash. If from `invoice+statements+acct_*@stripe.com` (real payment receipt), then label Financial.** The Granola/Stripe domain split matters.

- **If from a known recruiting platform (`@ashbyhq.com`, `@greenhouse-mail.io`, `@teamtailor-mail.com`, `@workday.com`, etc.) regardless of substantive content, then Hiring + auto-archive.** Adithya is in active job search; volume is high; he doesn't want individual notification noise.

- **If sender domain is a `.info`, `.us`, `.agency`, `.life`, `.work` TLD AND display name has "first last" pattern, then it's almost certainly a cold-outreach spam farm.** These get trashed. Real BD comes from company `.com`/`.xyz`/`.io` domains.

- **If subject contains a tracking hash like `[#cf96160de76d42ed92]` AND is from a `@google.com` subdomain, then it's a Google support ticket (e.g., cloudquota Quota Increase Request).** Goes to Dev (formerly Dev Tools).

- **Attachment types matter:** `.pdf` attachments often indicate contracts (Contracts) or invoices (Financial) or interview materials (Hiring). `.ics` attachments indicate calendar invites (TRASH). Image attachments (`image.png`, `D3531F64-6DB1...`) are screenshots in real conversations — preserve thread.

- **No strong body-length thresholds applied.** Heuristics are subject + sender driven.

- **CC/BCC patterns:** No specific rules applied. Multi-recipient threads (`Sarah, me, Rusty 8`) are flagged as conversation in the deserves-reply heuristic via thread-message-count, not directly via CC count.
