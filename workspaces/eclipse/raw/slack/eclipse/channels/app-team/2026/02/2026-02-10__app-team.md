> ## Analysis
>
> After analyzing the codebase and issue #54, I can see the problem clearly:
>
> ### Current Issue
> The Lambda migration handler (`lambda/migrate/handler.ts`) contains a hardcoded `MIGRATIONS_SQL` string that:
> 1. ***Is out of sync*** - Missing recent migrations like `20260120_add_api_key_model` and `20260130092805_add_transcription_pipeline`
> 2. ***Requires manual maintenance*** - Every new Prisma migration must be manually copied and transformed to idempotent SQL
> 3. ***No migration tracking*** - Creates the `_prisma_migrations` table but doesn't populate it, breaking Prisma's migration state tracking
> 4. ***Brittle approach*** - Concatenates all migrations into one block instead of tracking individual migration state
>
> ### Current Architecture
> - Main app uses Prisma CLI via `Dockerfile.migrate` and ECS tasks for migrations
> - Lambda migration handler exists as a workaround for VPC-only database access
> - Two parallel migration systems that can get out of sync
>
> ## Proposed Solution
>
> I'll implement the "Quick Fix" suggested in the issue comments with some improvements:
>
> ### Plan: Migration Sync Script + Automated Generation
>
> 1. ***Create a Node.js script*** (`scripts/generate-lambda-migrations.ts`) that:
>    - Reads all files from `prisma/migrations/*/migration.sql`
>    - Transforms non-idempotent SQL to idempotent forms
>    - Generates properly tracked migration records for `_prisma_migrations` table
>    - Outputs the result to `lambda/migrate/migrations.sql`
>
> 2. ***Transform SQL for idempotency***:
>    - Wrap `CREATE TYPE` with `DO $$ BEGIN ... EXCEPTION WHEN duplicate_object ...`
>    - Add `IF NOT EXISTS` to `CREATE TABLE`, `CREATE INDEX`, etc.
>    - Wrap `ALTER TABLE ADD CONSTRAINT` with conditional logic
>    - Handle `ALTER TABLE ADD COLUMN` with `IF NOT EXISTS` patterns
>
> 3. ***Proper migration tracking***:
>    - Generate INSERT statements for the `_prisma_migrations` table
>    - Include migration name, checksum, and completion timestamps
>    - This will keep Prisma CLI and Lambda migrations in sync
>
> 4. ***Update Lambda handler***:
>    - Load SQL from the generated file instead of hardcoded string
>    - Add proper transaction handling
>    - Better error reporting and logging
>
> 5. ***Add automation***:
>    - Update the script in `package.json`
>    - Add to CI/deployment pipeline to verify sync
>    - Could be added as pre-commit hook later
>
> ### Benefits of This Approach
> - :white_check_mark: ***Eliminates manual copy-paste errors***
> - :white_check_mark: ***Maintains single source of truth*** (Prisma migrations)
> - :white_check_mark: ***Proper migration state tracking***
> - :white_check_mark: ***Much simpler than architectural changes***
> - :white_check_mark: ***Backward compatible*** with existing deployment
> - :white_check_mark: ***Can be automated in CI***
>
> ### Files to Create/Modify
> 1. `scripts/generate-lambda-migrations.ts` - New migration generator script
> 2. `lambda/migrate/migrations.sql` - Generated SQL file
> 3. `lambda/migrate/handler.ts` - Updated to load from file and track migrations
> 4. `package.json` - Add script command
> 5. `.github/workflows/deploy-lambda.yml` - Add generation step
>
> This keeps the current architecture but makes it maintainable and reliable. The script approach is much simpler than switching to ECS/Fargate tasks or VPC access solutions, while solving the core sync and maintenance problems.
>
> ***Ready to proceed?*** I'll implement this solution step by step, creating the migration generation script first, then updating the Lambda handler, and finally updating the deployment pipeline.
