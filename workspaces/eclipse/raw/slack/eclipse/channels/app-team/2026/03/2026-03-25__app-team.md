**Title:** Separate API uploads from individual contributor submissions + hour tracking
**Description:**
We need to make the following changes to support the Fluffle payment and tracking workflow:
**Requirements:**
• **Separate pages/sections** for API uploads (Fluffle) vs. individual contributor submissions — currently everything is in one section
• **Track total hours submitted** per user, broken out by:
• API submissions
• Regular user submissions
• **Group conversations** by Fluffle serialized number (the identifier after `fluffle` before the `/` in the file name) to link two sides of the same conversation
• **Auto-generate invoices** every 2 weeks based on approved hours (bank transfer, not Stripe)
• **Do not trigger Stripe payments** on approval — payments will go through bank transfer on a biweekly cadence
**Context:**
• ~150 hours submitted so far
• Fluffle only sends conversations over 10 minutes
• We need to clearly surface how many hours from each user have been approved vs. not, so we can rack up totals for biweekly payouts
**Additional nice-to-have:**
• Add dialect/accent tags to metadata (they're working on providing this)

> Based on my analysis of the codebase and the issue requirements, here's my comprehensive plan to address issue #132:
>
> ## Analysis
>
> The current system has:
> 1. All uploads displayed together on the uploads page without separation
> 2. Basic hour tracking but no breakout by source (API vs manual uploads)
> 3. API uploads use `sourcePlatform: 'OTHER'` with `sourcePlatformOther: 'API: {apiKey.name}'`
> 4. No Fluffle serialized number grouping functionality
> 5. Stripe payments triggered on approval
> 6. No biweekly invoice generation system
>
> ## Proposed Plan
>
> ### 1. ***Separate API Uploads from Individual Submissions***
>
> ***Frontend Changes:***
> - Modify `/app/uploads/page.tsx` to add tabs/sections separating:
>   - ***"My Uploads"*** - Regular user submissions (`sourcePlatform: 'NONE'`)
>   - ***"API Uploads"*** - Partner uploads via API (`sourcePlatform: 'OTHER'` with API key attribution)
> - Add filters to allow toggling between views
> - Update the UI to clearly distinguish upload sources
>
> ***Database Changes:***
> - Add new enum value `FLUFFLE` to `SourcePlatform` in schema
> - Modify partner upload endpoint to use `FLUFFLE` instead of `OTHER` for Fluffle uploads specifically
>
> ### 2. ***Hour Tracking by Source***
>
> ***Backend Changes:***
> - Create new API endpoint `/api/user/stats/detailed` returning:
>   ```typescript
>   {
>     totalHours: number,
>     apiHours: number,
>     regularHours: number,
>     approvedApiHours: number,
>     approvedRegularHours: number,
>     pendingApiHours: number,
>     pendingRegularHours: number
>   }
>   ```
> - Update existing `/api/user/stats` to include breakdown
>
> ***Frontend Changes:***
> - Update stats display components to show hour breakdown
> - Add dashboard widgets showing API vs regular submission stats
>
> ### 3. ***Fluffle Serialized Number Grouping***
>
> ***Database Changes:***
> - Add `fluffleSerialNumber` field to Upload model
> - Add `conversationGroup` field to group related uploads
>
> ***Backend Changes:***
> - Extract Fluffle serial number from filename pattern in partner upload endpoint
> - Create API endpoint `/api/uploads/conversations` to group uploads by Fluffle serial number
> - Update partner upload endpoint to parse and store Fluffle serial numbers
>
> ***Frontend Changes:***
> - Add conversation view on uploads page showing grouped uploads
> - Allow expanding/collapsing conversation groups
>
> ### 4. ***Biweekly Invoice Generation***
>
> ***Backend Changes:***
> - Create new model `Invoice` in schema:
>   ```prisma
>   model Invoice {
>     id                String   @id @default(cuid())
>     userId            String
>     user              UserProfile @relation(fields: [userId], references: [id])
>     periodStartDate   DateTime
>     periodEndDate     DateTime
>     totalApiHours     Decimal  @db.Decimal(10, 2)
>     totalRegularHours Decimal  @db.Decimal(10, 2)
>     totalAmount       Decimal  @db.Decimal(10, 2)
>     status            InvoiceStatus @default(PENDING)
>     generatedAt       DateTime @default(now())
>     paidAt            DateTime?
>     uploads           Upload[] @relation("InvoiceUploads")
>   }
>
>   enum InvoiceStatus {
>     PENDING
>     SENT
>     PAID
>     CANCELED
>   }
>   ```
> - Add `/api/internal/invoices` endpoint for generating and managing invoices
> - Create background job for automatic biweekly invoice generation
>
> ### 5. ***Disable Stripe Auto-Payments***
>
> ***Backend Changes:***
> - Modify review approval logic to NOT trigger Stripe payments
> - Add configuration flag to control payment method (Stripe vs bank transfer)
> - Update payout logic to mark uploads as "approved for bank transfer"
>
> ### 6. ***Nice-to-have: Dialect/Accent Tags***
>
> ***Database Changes:***
> - Add `dialect` and `accent` fields to Upload model
> - Update partner upload metadata to accept these fields
>
> ---

> 1. ***Database Schema Updates*** - Add new fields and models
> 2. ***Backend API Changes*** - Create new endpoints and modify existing ones

> 4. ***Invoice System*** - Build invoice generation and management
