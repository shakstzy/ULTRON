> your intuition about reduced context = better agent performance is backed by real data. microsoft's azure SRE agent team found that agent quality drops non-linearly as you add more context/tools — their conclusion was literally "give the model fewer, cleaner choices." anthropic's context engineering docs say the same thing. a focused audio data API with a narrow surface area is what agents are optimized for. i'm literally seeing this play out with openclaw right now lol
>
> on the smaller buyer thing — nobody is serving this segment. [Defined.ai](http://Defined.ai), Innodata, Opendatabay are all enterprise-first with $50K+ minimums. but there's a massive long tail of vertical AI startups, fine-tuning shops, and non-US voice companies who need like 50-500 hours of niche audio. AI training data market is ~$7.5B this year heading to $52B by 2035. we don't need the whole pie — just the consent-verified self-serve slice i need to read more on this cuz theres def yc startups and shit we can target that might be in this segment.
>
> re: agent-native — the infrastructure is literally being built for us right now. MCP is at 97M+ monthly SDK downloads, adopted by every major lab. the MCP registry ("app store" for agent tools) is on the 2026 roadmap. Google just launched Universal Commerce Protocol for agentic commerce, OpenAI built an Agentic Commerce Protocol w Stripe, and Stripe shipped an entire Agentic Commerce Suite for getting "discovered by AI agents." McKinsey has agentic commerce at $3-5T globally by 2030, but like we should def align w what the most common denominator is and in this case thats mcp
>
> the "forwarding work" idea is the killer move — if we expose our catalog as an MCP server, any higher-level platform (Claude, ChatGPT, Perplexity, custom enterprise agents) can route audio data requests to us programmatically. we become THE canonical audio data tool in the agent ecosystem
>
> on sequencing: start w self-serve API for human buyers (validates demand, generates revenue, forces us to build the API layer), then layer agent-native endpoints on top. the API work is the same — difference is just who's calling it. once we're live as an MCP server the forwarding happens naturally
>
> and the consent provenance + SOC 2 path = the trust layer that makes agents pick us over alternatives. agents optimize for reliability and verifiability, not brand. that's our moat
