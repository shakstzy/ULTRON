# graphify notes — personal workspace

These are workspace-specific runbook hints for `/graphify` on `personal/`. The graphify SKILL reads this file (Step 1.5) when it sees `<INPUT_PATH>/.graphify-notes.md` or `<INPUT_PATH>/../.graphify-notes.md`. Don't surface it to the user unless asked — just absorb the guidance.

## Skip these dirs (extraction would burn tokens for ~zero signal)

- `raw/imessage/_profiles/` — 1,055+ stub files, median 21 words, max 160. Frontmatter-only contact pointers. Treat each stub as a cheap "this person exists" node (id = filename slug, dashes→underscores) and DO NOT route to a semantic subagent.

## Chunking + dispatch

- Files are tiny markdown — median ~1.2KB, mean ~2.8KB, p99 ~27KB.
- Safe chunk size: **100 files/chunk** (was 50 last run; subagents had headroom).
- Sort files by directory so per-contact files cluster (alpha-sort approximates this since paths embed contact slug).
- Dispatch ALL chunks in one message — runtime parallelizes anyway. Sequential waves of 16 cost ~10 min wall time for nothing.
- Use `subagent_type="general-purpose"` (must have Write access). Never `Explore` — its results never land on disk and chunks vanish silently.
- If reviewer critique applies (Codex was strict on this): keep chunks small enough that a chunk failure is recoverable. 100 files is the safe upper bound.

## Place-hub dupes still need post-merge

Subagents drift on labels for places (no shared id source). After Part C merge, fuzzy-merge place hubs:

- `austin` ≡ `austin_tx`, `los_angeles` ≡ `la`, `new_york_city` ≡ `nyc`, `san_francisco` ≡ `sf`, etc.
- `dad` ≡ `dad_NNNN` (and similar `_NNNN` phone-suffix variants for any contact node)

A simple heuristic that works: if two node ids share the same alpha prefix AND one ends in `_NNNN` (4-digit phone suffix) AND no edges contradict, merge into the suffix-less id.

**Person-name dupes are NOT a post-merge problem anymore** — the SKILL subagent prompt now refuses to emit person-merge edges without hard evidence (shared phone/email/handle/contact_slug or thread continuity). Trust the upstream filter; don't add post-process heuristics that re-introduce the problem.

## AMBIGUOUS audit trail

The SKILL now drops AMBIGUOUS edges by default and writes them to `graphify-out/AMBIGUOUS_REJECTS.jsonl` for review. After the run, optionally skim that file. If a real pattern of useful edges shows up there, the upstream subagent prompt needs another tightening pass — but don't re-enable AMBIGUOUS on this workspace based on one-off finds.

## Token tracking is broken — ignore cost.json

Subagents return `"input_tokens": 0, "output_tokens": 0` because they don't track usage. The orchestrating Claude sees real usage in `<usage>` blocks of each Agent tool result, but those aren't captured into the chunk JSON. Real cost on the 2026-05-06 run was ~5–10M tokens of subagent context; cost.json reported 18k/3.5k. Don't trust that file. (Fix candidate: have orchestrator parse `<usage>` from each Agent result and write to a per-chunk metadata sidecar.)

## Cron is currently broken — manual `/graphify` only

`com.adithya.ultron.graphify-personal.plist` calls `run-stage.sh graphify personal`, but `graphify` is not a registered stage in `_shell/bin/run-stage.sh`. Cron exits 0 (graceful skip) and silently does nothing. Until that's fixed, don't expect daily auto-refresh — graph staleness is on the user.

## --update vs full rebuild

For incremental refresh (after the cron is fixed), use `/graphify personal --update` instead of full rebuild. The --update path re-extracts only changed/new files and merges into existing graph. Full rebuild on 4k iMessage files = 80 subagents = ~$15-30 in tokens. --update on a typical day = handful of new monthly files = 1-2 subagents.

## Workspace-specific extraction hints

For iMessage corpus, the per-file frontmatter is rich:
- `contact_slug` → use as node id base (dashes→underscores). This is the ONLY reliable identity key — same slug across files means same person; different slug means different person, period. Do not collapse on label similarity.
- `contact_type` → `individual` or `group` — feed to node attribute
- Slug suffixes encode location/context: `-nyc`, `-la`, `-uc-berkeley`, `-hinge-austin`, etc. — INFERRED `lives_in` / `studies_at` edges OK at confidence 0.8
- `_profiles/` (per skip rule above) — stub-only, but the slug itself names a contact; create the node from the filename even when skipping semantic extraction.

The personal workspace was recently scoped down (per its `CLAUDE.md`): music, health, dating, finance, trading, real-estate, property-management, library each have their own workspaces now. So `personal/raw/imessage/` is filtered to truly-personal contacts (friends, family, household), not the full iMessage archive. The corpus may have shrunk since the 2026-05-06 run.
